# Рабочий контекст по проекту `metamath2py`

## 1. Зачем существует этот репозиторий

Исходная цель, описанная в `artifacts/main.pdf` и исходниках статьи в `paper/`, состоит в том, чтобы превратить корпус Metamath в более удобное для LLM и Python-экосистемы представление:

- брать теоремы и доказательства из `set.mm`;
- переводить их в исполняемые Python-классы;
- использовать это как датасет для prompting, fine-tuning и пошагового вывода.

Текущая цель проекта уже шире статьи:

- на входе получать цель на естественном языке;
- строить формализацию этой цели в терминах Metamath;
- строить или восстанавливать доказательство;
- проверять промежуточный Python-артефакт локальным верификатором;
- в пределе уметь добавлять новую теорему и доказательство в `set.mm` так, чтобы исходная программа Metamath из `metamath_program/` успешно это верифицировала.

Иными словами, статья описывает слой "Metamath -> Python dataset", а текущий репозиторий движется к слою "Natural language -> theorem state -> proof state -> Metamath-verifiable result".

## 2. Что здесь считать источником истины

При работе с этим репозиторием полезно держать следующую иерархию:

- `artifacts/main.pdf` и `paper/*.tex` объясняют исходную идеологию и мотивацию проекта.
- `artifacts/metamath.pdf` это книга/справочник по самому Metamath; использовать как внешний источник истины по синтаксису, структуре доказательств и общему устройству системы.
- `metamath_program/metamath/set.mm` и `metamath_program/metamath/metamath(.exe)` это финальный критерий корректности на уровне оригинального Metamath.
- `verification.py` это локальный критерий корректности на уровне Python-представления доказательства.
- существующий корпус теорем надо считать базой знаний и опорой для retrieval/search, а не тем, что нужно произвольно переписывать.

## 3. Как проект сейчас распадается на слои

### Слой A. Перевод Metamath в Python и сборка датасета

- `mm.py`, `models/` и `code_builders/` содержат переписанную логику на базе `mmverify.py`: парсинг Metamath, обработку фреймов, подстановок и генерацию Python-классов.
- `build_jsonl_dataset.py` делает JSONL-датасет из `set_normal.mm`.
- `build_dataset_of_python_files.py` превращает JSONL в набор Python-файлов с theorem/proof модулями.
- `verify_metamath2py_files.py` и `verification.py` проверяют такие сгенерированные proof-модули.

### Слой B. Восстановление или извлечение теорем по описанию

- `database/` поднимает OpenSearch-индекс по имеющемуся корпусу theorem/proof файлов.
- `metamath_agent/offline_runner.py` умеет по описанию и имени теоремы либо вытащить артефакты через retrieval, либо восстановить их напрямую из Metamath-источника.
- Это пока в основном режим "найти/восстановить уже существующее знание", а не полноценное изобретение новой теоремы.

### Слой C. Поиск по дереву состояний теоремы и доказательства

- `saplings/` это наиболее важный слой для текущей долгосрочной цели.
- Здесь цель на естественном языке хранится как `CreateNodeTask`, внутри которого есть `TheoremState` и `ProofState`.
- `saplings/saplings_agents/candidate_generator.py` генерирует patch-наборы для теоремы и доказательства.
- `saplings/saplings_agents/a_star.py` делает поиск по дереву.
- `saplings/node_scorer.py` оценивает узлы по двум главным сигналам: прогресс локальной верификации и структурный прогресс доказательства.
- `saplings/tools/theorem_recovery.py` умеет материализовать текущее состояние узла во временный theorem/proof код и прогнать верификацию.

### Слой D. Веб-визуализация и ручное наблюдение за поиском

- `web/` даёт локальный UI для просмотра графа поиска.
- `web/app.py` поднимает Flask-приложение.
- `web/state.py` хранит состояние A* поиска, фронтир, узлы, рёбра и сериализацию/десериализацию сессии.
- `web/session_config.json` задаёт пример цели и параметры запуска.

### Слой E. Оригинальный Metamath и финальная верификация

- `metamath_program/metamath/` содержит исходную программу Metamath, её C-исходники, бинарники и базы `set.mm`, `set_normal.mm` и другие `.mm`.
- `artifacts/verification_using_program_example.md` показывает базовый сценарий проверки `set.mm` через исходную программу.
- `metamath_adapter.py` и связанные утилиты используются как мост к CLI Metamath.

## 4. Что лежит в основных папках

- `artifacts/`:
  статья в PDF, книга по Metamath и пример ручной верификации через оригинальную программу.
- `paper/`:
  исходники статьи; читать удобнее, чем PDF, если нужно быстро понять идеологию.
- `code_builders/`:
  генерация Python-классов и proof-кода из распарсенного Metamath.
- `models/`:
  базовые структуры парсера и верификатора Metamath-уровня.
- `database/`:
  OpenSearch-слой для retrieval по корпусу theorem/proof файлов.
- `examples/`:
  небольшие готовые примеры theorem/proof модулей.
- `metamath_agent/`:
  CLI и промежуточные сценарии восстановления/агентного получения theorem/proof артефактов из описания.
- `saplings/`:
  дерево поиска, patch-операции, scoring, theorem recovery и связанная агентная логика.
- `web/`:
  визуализация графа поиска и логирование шагов.
- `metamath_program/`:
  оригинальная программа Metamath и файлы баз знаний.
- `tests/`:
  регрессии для локального верификатора, node scoring, retrieval и восстановления теорем по описанию.
- `out/`:
  выходные артефакты и логи.
- `agent_tasks/`:
  заметки, задачи и вспомогательные документы для агентной работы.

## 5. Практически важные замечания

- В `.gitignore` исключён каталог `/metamath2py/`. Это значит, что ожидаемый runtime-пакет со сгенерированными `classes/` и `proofs/` может существовать локально, но отсутствовать в git.
- Текущий checkout содержит `examples/classes` и `examples/proofs`, а также почти пустой корневой `classes/`. Поэтому при задачах, связанных с генерацией и верификацией, надо отдельно проверять, куда именно код сейчас пишет theorem/proof модули.
- `paths.py` и `llm_authoring.py` ориентируются на пакет `metamath2py/...`, то есть на сгенерированную структуру, а не на корневые `examples/`.
- Статья не описывает полностью текущий поиск по дереву. Для современных задач по NL->proof важнее смотреть в `saplings/`, `metamath_agent/`, `web/` и соответствующие тесты.

## 6. Какой рабочий mental model использовать дальше

Если приходит новая задача по этому репозиторию, полезно думать так:

1. Есть неизменяемый формальный корпус Metamath и оригинальный верификатор.
2. Есть слой перевода этого корпуса в Python-представление, удобное для LLM.
3. Есть retrieval по готовым теоремам и примерам.
4. Есть поиск по дереву состояний, который постепенно уточняет theorem state и proof state.
5. Любой хороший кандидат надо уметь:
   сначала проверить через `verification.py`,
   а в идеальном случае затем довести до артефакта, который встраивается в `set.mm` и проходит проверку исходной программой Metamath.

## 7. Что читать в первую очередь при следующем заходе

- `agent_tasks/project_context.md` — этот файл.
- `paper/abstract.tex`, `paper/translation.tex`, `paper/conclusion.tex` — чтобы быстро вспомнить исходную идею.
- `saplings/saplings_agents/candidate_generator.py` — чтобы понять, как сейчас создаются новые кандидаты.
- `saplings/node_scorer.py` и `saplings/tools/theorem_recovery.py` — чтобы понять, как кандидаты оцениваются и материализуются.
- `metamath_agent/agentic_runner.py` и `web/app.py` — чтобы понять текущие точки входа для поиска.
- `verification.py` и `artifacts/verification_using_program_example.md` — чтобы не забывать разницу между Python-верификацией и проверкой оригинальным Metamath.
